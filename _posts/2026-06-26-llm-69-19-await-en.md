# We Moved from EC2 to ECS, and Response Time Jumped to 69 Seconds

In [my previous post](https://velog.io/@5hseok/esc-migration-from-ec2) I wrote, "I'll cover why the earlier migration failed and how I got it working in a follow-up post." This is that post.

Here's what happened. We moved a background worker that **generates LLM responses with LangChain** from EC2 to ECS, and a job that normally finished in **3.5 seconds** stretched out to **up to 69 seconds** under traffic. We ended up rolling back to EC2 to stop the bleeding. When the migration landed back on my desk, I decided to dig into "so what exactly went wrong back then" from scratch.

The cause turned out to be a bit unexpected. **It wasn't the LLM, it wasn't the monitoring agent, and it wasn't the CPU.** The code we were certain ran asynchronously — because we'd put `await` on it — was actually running synchronously under the hood. This post is a troubleshooting log of how I tracked down that cause, and at the same time it unpacks the idea that **"using `async`/`await` doesn't mean your code is actually async."**

> Written for intermediate backend developers. If you've heard of asyncio and threads but want to know "okay, but how does that actually blow up in production?", this should be a good fit.

# What Actually Happened

Let's start with the measured numbers.

The fact that scaling out the workers made things *worse* was the first clue. Normally adding more workers should speed things up, but it got slower. That's a signal that the bottleneck is **structural**, not a matter of insufficient resources.

And there was a real mystery on top of that. When I broke the job down stage by stage and timed each part, this is what I got.

The LLM call was a healthy 4.6 seconds, and the DB write was 0 seconds. Yet the whole job took 69 seconds. **Where did the other 64 seconds go?** No single stage inside the job ate 64 seconds, but added together it became 69. Something in the middle was eating all the time.

# Eliminating Suspected Causes One by One

When something is "slow," there's a usual list of suspects. Instead of waving them off with guesses, I **confirmed every one of them against real measurements and ruled them out.** This elimination process is what gives the conclusion its credibility.

![표 2](https://media.vlpt.us/images/5hseok/post/b645498f-da04-4992-8718-0ac7b6145b50/table2_zfxxjvp2.png)


There was one more. At first I strongly suspected the **monitoring agent (APM)**. When we moved to ECS, the monitoring agent's connection model also changed (from one per container to one per host). The timing lined up suspiciously well, so I thought, "this is it."

But when I ran a controlled load test — **the slowdown stayed exactly the same even with the monitoring agent fully turned off.** In fact it sometimes got worse. The most likely culprit I'd been eyeing turned out to be wrong.

![다이어그램 2](https://media.vlpt.us/images/5hseok/post/c0abc915-fc84-4df4-90cf-44c38f59d647/diagram2_n6vcfc6x.png)

After subtracting everything I'd ruled out, only one thing remained logically. **The same image (the same code), yet EC2 is permanently fine and only ECS breaks.** Resources all have headroom. So it's neither the code's fault nor a resource shortage — it means **"something that behaves differently depending on the environment"** is hiding in the code.

To understand that something, we need to pause on one concept.

---

# Hold On — Putting `await` On It Doesn't Make It Async

This is the core concept of the whole problem. Our worker code clearly looked like this.

```python
async for chunk in llm.astream(messages):
    ...
```

We put `await` (well, `async for`) on `astream`, so naturally we believed it was async. But it wasn't. To understand why, you have to walk through three things in order.

## ① GIL — A Python Thread Runs Only One Line at a Time

Python (CPython, to be precise) has a lock called the **GIL (Global Interpreter Lock)**. Within a single process, the thread actually executing Python code is **always exactly one**. Whether you have 10 threads or 100, it's the same.

But there's an important exception. **The GIL is only held "while executing Python code," and it's released while waiting on I/O.**

![표 3](https://media.vlpt.us/images/5hseok/post/6526831f-ce04-445a-ba34-15cabe15b5dd/table3_l40dvzpz.png)


Our worker's job is **waiting for an LLM response** — extreme I/O bound. 99% of the work is "waiting on an external response." This is exactly why, later on, "the fix is expanding threads, not adding CPU." For now, just keep it in mind.

## ② Event Loop — One Waiter Handles 20 Tables

`asyncio` runs hundreds of coroutines in turns **on a single thread**. The thing that directs that traffic is the event loop. The restaurant-waiter analogy fits perfectly.

One waiter (the event loop) handles 20 tables (20 jobs). They don't stand frozen at one table waiting for the food. They take the order, hand it to the kitchen, and immediately move to the next table. That "time waiting for the food" is the I/O wait. So **even a single waiter appears to be progressing all 20 tables at once.**

```python
async def handle_table(table):
    order = take_order(table)         # I do this myself
    food = await kitchen.cook(order)  # ← "yield" here. Go check other tables
    serve(table, food)                # come back and continue when the food is ready
```

The key rule is this. **Because every task yields with `await` while it waits,** they appear to progress concurrently. But what if some code just blocks without `await`? **The waiter freezes at that table.** All the other 19 tables stop dead.

## ③ How to Slot Synchronous Code into the Async World — `run_in_executor`

The problem is that a huge number of libraries in the world are still **synchronous (blocking)**. The poster child is the official AWS SDK, **`boto3`**. When you call an external API with `boto3`, it holds the entire thread until the response comes back. There's no `await`.

If you call this directly on the event loop thread, the waiter freezes. So asyncio **hands this kind of synchronous code off to a separate worker thread.** The tool for that is `run_in_executor`.

![다이어그램 3](https://media.vlpt.us/images/5hseok/post/7579dec6-59a6-4137-9dc6-b71c899890eb/diagram3_pzh9lyq9.png)

Here's where the **executor** enters. executor = "a pool of helper threads that take on blocking work." And if you don't pass one explicitly (the default executor), the number of helper threads is decided like this:

```
max_workers = min(32, (number of CPU cores) + 4)
```

**This single line was the whole story.** On a 2 vCPU instance, `min(32, 2+4)` = **6**. Even on 4 vCPU, `min(32, 4+4)` = **8**. That's only 6–8 helper threads.

---

# 🔥 The Real Cause — LangChain `ChatBedrockConverse` and Thread Starvation

Now let's put the pieces together. The LLM client we used was `langchain`'s `ChatBedrockConverse`. But this class internally **uses the synchronous SDK `boto3`.** It has no real async implementation (`_astream`).

The langchain base class has an automatic fallback: "if there's no real async implementation, wrap the synchronous implementation in `run_in_executor` and pretend it's async." So even though we clearly wrote `async for ... astream(...)`, **it was async only in syntax; underneath it was a synchronous call eating one helper thread.**

![다이어그램 4](https://media.vlpt.us/images/5hseok/post/4fab3df5-8645-4f3a-8ba7-865303274c2a/diagram4_munao0p8.png)

Now the collision is visible. A worker runs **20 jobs concurrently** in one process (concurrency cap `MAX_JOBS=20`). All 20 try to call the LLM. Each one demands a helper thread, but **there are only 6 helper threads.**

![다이어그램 5](https://media.vlpt.us/images/5hseok/post/b4ddb1b5-89b3-4ea7-97a2-c8e9a18a596b/diagram5_efqnvuee.png)

This is **thread starvation**. The jobs want to call the LLM, but **with no free helper thread to call from**, they sit queued inside the worker without even sending the external request. That's why the metrics showed this contradiction.

- **The LLM was fast** — judged only by when the request arrived, 2–3s. True. The request just arrived late.
- **The CPU was idle (13%)** — because it was "waiting for a thread slot," not computing.
- **Yet the whole job took 69 seconds** — all of the time leaked into waiting for a thread slot.

That's the answer to the opening question, "where did the other 64 seconds go?" It leaked neither inside the LLM nor inside the DB, but in **"the queue waiting for a thread slot."**

---

# But Why Was EC2 Fine?

Here's something that doesn't add up. Same code, same libraries, yet on EC2 it ran fine for months. If the defect were in the code, EC2 should have blown up too. What was different?

The answer was **not the spec of a single instance, but "the total number of processes."**

The worker spins up 2 processes per host (instance), and each process builds an executor by the formula above. But EC2 and ECS differed in "how many machines the same amount of work is split across."

![표 4](https://media.vlpt.us/images/5hseok/post/b27438c4-3706-4217-8a3b-22f50ca6b062/table4_gs1edmh9.png)


See the twist? Looking at a single instance, ECS has more CPU (4 vCPU) and a bigger executor (8 > 6). **And it still starved.** Because going "fewer big instances" cut the total processes in half, 8 → 4. When the same amount of work is funneled into half the processes, **concurrent jobs per process double.**

![다이어그램 6](https://media.vlpt.us/images/5hseok/post/624c84d3-5336-4fa9-b079-589deeaeea88/diagram6_0np80e80.png)

EC2's 8 processes absorbed the load in fine-grained slices, keeping concurrent jobs per process around 3. That fits inside the 6-thread executor, so there was headroom. **The latent defect had been masked by the generous process count.**

ECS has 0 headroom, so it falls into a vicious cycle the moment a job gets even slightly slower. A job that can't grab a thread gets slower, the slower job holds its slot longer, concurrent jobs climb toward `MAX_JOBS=20`, and 8 threads can't handle 20, so response time keeps growing. That's also why it spiked not right after the ECS move, but **the moment traffic accumulated and hit the threshold.**

> "Fewer big instances" can be a loss compared to "many small instances" from a thread perspective. We looked only at instance specs and figured "it's bigger, so it'll be fine," missing the thing that actually mattered — **the total process count** dropping.

---

# The Fix — Grow the Executor Directly (Not the CPU)

Since the cause is "6 helper threads < 20 concurrent jobs," the fix is simple. **Increase the number of helper threads explicitly.** At worker startup, swap the default executor for a bigger one.

```python
# worker startup
asyncio.get_running_loop().set_default_executor(
    ThreadPoolExecutor(
        max_workers=40,            # MAX_JOBS(20) × 2, 2x headroom
        thread_name_prefix="llm-exec",
    )
)
```

The load-test results were clear.

![표 5](https://media.vlpt.us/images/5hseok/post/17980b63-5cd8-4ee0-8fee-c7fcc348d5da/table5_a4hx4a1p.png)


One thing worth calling out here. **Why expand threads instead of adding CPU?**

![다이어그램 7](https://media.vlpt.us/images/5hseok/post/648c9605-fedc-45dc-91ba-f3ab788c0079/diagram7_yvg77l8g.png)

Even with 40 threads, most are **in an I/O wait state** and have released the GIL (remember ① above). So they barely consume CPU. The only extra cost is a little memory per thread, and **no instance upgrade was needed.**

> ⚠️ That said, "more threads = better" is not a universal rule. If this had been **CPU bound** work, adding threads wouldn't help because of the GIL — it would only add context-switching overhead. This was a case where **thread expansion is the answer precisely because it's I/O bound.** So judging the nature of the work (I/O vs CPU) comes first.

---

# But Is This Really the Root Fix — Let Me Be Honest

It looks like a clean ending up to here, but honestly, growing the executor **stopped the symptom, not cured the disease.** The real disease is that `boto3` is a synchronous SDK, so LLM calls only run by borrowing a helper thread. No matter how big you make the thread pool, the structure of wrapping a synchronous call in a thread remains.

So what's the real end? `ChatBedrockConverse` having a real async implementation (`_astream`). Then the event loop handles LLM calls directly without borrowing helper threads, and you'd never have to think about executor size at all.

So I thought, "then should I implement `_astream` and open a PR to LangChain?" and dug into the code — only to find the answer already written there. **LangChain's stance is "if `boto3` supports native async, then we'll support async streaming too."** But `boto3`'s native async support has made no progress for years. In the end LangChain is just waiting on `boto3`, and waiting for that felt endless.

So I leaned toward "fine, I'll just open an issue and contribute the implementation myself." ...except here's a slightly deflating story.

I've actually tried something like this once before. A while back, to log the request-id of LLM responses (so I could jump straight to the provider console when debugging), I opened [an issue proposing the feature](https://github.com/langchain-ai/langchain/issues/36983) on LangChain. There was a prior PR that had finished the implementation but got closed after the contributor went silent, so I raised my hand to take it over and carry it to merge. And that issue — **it's been almost two months and there's still no response from a maintainer.** LangChain has a policy of "don't start PR work until you're assigned," so there's nothing to do but wait for an answer.

...which, honestly, makes me think the async streaming side would go the same way — "open an issue and wait two months again?" — and the motivation just isn't there. ~~So I've been putting it off.~~ I'll get to it. Probably.

Still, the direction is clear. Growing the executor is a stopgap that keeps the worker alive right now, and the real end is native async landing upstream. Someday when that gets merged, the day will also come when this post's one line of `set_default_executor` can be deleted.

# Conclusion — Just Remember This

It was a long detour, but the conclusion is one sentence.

> **The slow part was neither the LLM nor the monitoring — it was that our worker's thread pool, unable to fire the synchronous (boto3) calls asynchronously, was smaller than the number of concurrent jobs, so it couldn't even send the external request (thread starvation).**

Distilled into five points:

1. **Seeing `async`/`await` syntax doesn't mean it's async.** Whether the underlying SDK is natively async, or a synchronous SDK wrapped in a thread — that's the fork in the road. `boto3` is the latter.
2. Synchronous calls run on helper threads via `run_in_executor`, and the **default executor is `min(32, cpu+4)`** — smaller than you'd think.
3. When **concurrent jobs > executor size**, you get thread starvation where you can't even fire the external request and end up queued.
4. That's why the LLM, CPU, and DB are all fast yet **the whole job slows down** — the time leaks into "waiting for a thread slot."
5. An I/O bound bottleneck is solved by **expanding threads (executor), not adding CPU.** Judging the nature of the work comes first.

And the trick to ultimately narrowing the cause was **ruling out each suspected cause with real measurements** instead of fixing by guesswork. Without the process of directly ruling out the monitoring agent — the most plausible-looking suspect — I'd have wasted more time fixing the wrong place.

## One More Thing

Separately from this problem, there was another phenomenon where responses occasionally stretched out in **sharp 15-second increments**. At first I wondered if this was also thread starvation, but digging through the logs showed an entirely different cause — a delay from the model refusing the response and going through a fallback chain. That's unrelated to the thread pool, a matter of the model/prompt domain. I'll cover that one separately in another post.

> One more lesson. The belief that "I wrote it async, so concurrency is fine" only holds **when every call inside is genuinely async.** The moment one synchronous SDK slips in, an invisible thread pool becomes the ceiling on your whole concurrency.
